# design-doc

このスキルは対話形式でヒアリングを進めながら、機能設計書・画面設計書・API設計書・DB設計書を順番に作成し、`docs/` 配下に保存します。

---

## Step 1: 機能の概要ヒアリング

**1つのメッセージ**で以下を質問する。すでに伝えられている情報はスキップする。

1. **機能名**（日本語の表示名 と 英語のファイル名用スラッグ）  
   例: 「注文管理 / orders」
2. **誰が使うか**（ユーザーロール）  
   例: 「管理者のみ」「ログイン済みユーザー全員」「未認証でも使える」
3. **何を実現したいか**（この機能の目的・背景）  
   例: 「顧客からの注文を受け付け・管理できるようにしたい」
4. **主要な操作**（必要な CRUD やそれ以外の操作）  
   例: 「一覧・登録・編集・削除」「CSV インポートも必要」
5. **スコープ外（今回やらないこと）**  
   例: 「支払い処理は含まない」「未定ならスキップ可」

---

## Step 2: 機能設計書の作成

ヒアリング結果をもとに機能設計書の内容を作成し、ユーザーに確認を取ってから `docs/features/<slug>/<slug>.md` として保存する。

### 確認の取り方

設計書の内容をチャット上に表示し、「この内容で機能設計書を保存してよいですか？修正があればお知らせください。」と聞く。OKが出たらファイルを保存する。

### ファイル内容のガイドライン

```markdown
# 機能設計書: <機能名>

| 項目 | 内容 |
| --- | --- |
| 作成日 | <今日の日付> |
| 更新日 | <今日の日付> |
| ステータス | 草案 |

---

## 概要

<ヒアリングの「目的・背景」をもとに2〜3文で説明>

---

## ユーザーストーリー

<ヒアリング内容から1つ以上のユーザーストーリーを「〇〇として、〜のために、〜したい」形式で記載>

---

## 機能要件

### 必須（Must）

<主要な操作から Must 要件を箇条書き>

### あると望ましい（Should）

<あれば記載、なければ省略>

### 対象外（Won't）

<スコープ外として挙げられた内容を記載>

---

## 受け入れ条件

<機能要件を「〇〇した場合、△△される」という具体的なチェックリスト形式で記載>

---

## 画面・遷移の概要

<必要と思われる画面を箇条書きで列挙。詳細は後続の画面設計書で定義する>

---

## 関連ドキュメント

- 画面設計書: `docs/screens/<slug>/*.md`
- API設計書: `docs/api/<slug>/<slug>.md`
- DB設計書: `docs/database/<slug>/<slug>.md`
```

---

## Step 3: 画面のヒアリング

機能設計書を保存したら、画面について**1つのメッセージ**で追加ヒアリングする。

1. **必要な画面の一覧**  
   例: 「一覧・詳細・登録・編集の4画面」「一覧と登録だけでよい」
2. **各画面の主要な要素**（テーブル、フォーム、ボタン、検索条件など）  
   例: 「一覧画面はキーワード検索＋テーブル表示、詳細画面は削除ボタンつき」
3. **権限による表示の違い**（あれば）  
   例: 「削除ボタンは管理者のみ表示」

機能設計書の「画面・遷移の概要」が十分に詳細であれば、追加質問を省略してよい。

---

## Step 4: 画面設計書の作成

各画面ごとに設計書を作成し、ユーザーに確認を取ってから `docs/screens/<slug>/<screen>.md` として保存する。

画面が複数ある場合は全画面分をまとめてチャット上に提示し、「これらの画面設計書を保存してよいですか？」と確認してから一括保存する。

### ファイル名の例

| 画面 | ファイル名 |
| --- | --- |
| 一覧 | `docs/screens/orders/list.md` |
| 詳細 | `docs/screens/orders/detail.md` |
| 登録 | `docs/screens/orders/create.md` |
| 編集 | `docs/screens/orders/edit.md` |

### ファイル内容のガイドライン

```markdown
# 画面設計書: <画面名>

| 項目 | 内容 |
| --- | --- |
| 作成日 | <今日の日付> |
| 更新日 | <今日の日付> |
| ステータス | 草案 |
| パス | `/<slug>` または `/<slug>/:id` など |
| 認証 | 不要 / 要認証 / 管理者のみ |

---

## 概要

<この画面の目的と対象ユーザーを1〜2文で>

---

## 画面構成（ワイヤーフレーム）

<テキストアートでざっくりしたレイアウトを表現>

---

## コンポーネント一覧

<主要なUIパーツをテーブルで列挙>

---

## 状態と表示

<データあり・なし・ローディング・エラーなどの状態別の表示をテーブルで>

---

## ユーザー操作と挙動

<操作→挙動の対応をテーブルで>

---

## 画面遷移

<この画面から遷移できる先をテキストで>

---

## 使用するAPI

<この画面で呼び出すAPIのメソッド・エンドポイント・用途をテーブルで>

---

## 関連ドキュメント

- 機能設計書: `docs/features/<slug>/<slug>.md`
- API設計書: `docs/api/<slug>/<slug>.md`
```

---

## Step 5: API設計書の作成

機能設計書・画面設計書から必要なAPIを導出し、追加でヒアリングが必要な点があれば質問する。通常以下の点を確認する。

- 一覧APIにページネーションは必要か
- 検索・フィルタ条件のフィールド
- レスポンスに含めるべき追加フィールド

ユーザーに確認を取ってから `docs/api/<slug>/<slug>.md` として保存する。

### ファイル内容のガイドライン

```markdown
# API設計書: <機能名> API

| 項目 | 内容 |
| --- | --- |
| 作成日 | <今日の日付> |
| 更新日 | <今日の日付> |
| ステータス | 草案 |
| ベースパス | `/api/<slug>` |

---

## エンドポイント一覧

<機能要件・画面設計から導出したエンドポイントのテーブル>

---

## エンドポイント詳細

<各エンドポイントのリクエスト・レスポンス仕様を詳細に記載>

---

## エラーレスポンス共通仕様

| ステータス | 説明 |
| --- | --- |
| 400 | バリデーションエラー |
| 401 | 未認証 |
| 403 | 権限不足 |
| 404 | リソース未存在 |
| 500 | サーバーエラー |

---

## 関連ドキュメント

- 機能設計書: `docs/features/<slug>/<slug>.md`
- DB設計書: `docs/database/<slug>/<slug>.md`
```

---

## Step 6: DB設計書の作成

これまでのヒアリング・設計書から必要なテーブルとカラムを導出する。追加でヒアリングが必要な点があれば質問する。通常以下の点を確認する。

- 主要フィールドの名前・型・制約（NOT NULL など）
- 他テーブルとのリレーション（外部キー）
- 特殊なインデックスが必要か

ユーザーに確認を取ってから `docs/database/<slug>/<slug>.md` として保存する。

### ファイル内容のガイドライン

```markdown
# DB設計書: <テーブル名>

| 項目 | 内容 |
| --- | --- |
| 作成日 | <今日の日付> |
| 更新日 | <今日の日付> |
| ステータス | 草案 |
| テーブル名 | `<slug>` |

---

## 概要

<このテーブルが管理するデータと目的>

---

## テーブル定義

<カラム名・型・NULL制約・デフォルト値・説明のテーブル>
必ず id・created_at・updated_at・deleted_at（論理削除用）を含める。

---

## インデックス

<インデックスのテーブル>

---

## 外部キー制約

<外部キーがあればテーブルで記載>

---

## リレーション

<他テーブルとの関係をテキストで表現>

---

## Drizzleスキーマ（参考）

<Drizzle ORMのスキーマ定義コードを記載>

---

## 関連ドキュメント

- 機能設計書: `docs/features/<slug>/<slug>.md`
- API設計書: `docs/api/<slug>/<slug>.md`
```

---

## Step 7: 完了報告

全ドキュメントの保存が完了したら以下の形式で報告する。

```
設計書作成完了: <機能名>

作成したファイル:
  docs/features/<slug>/<slug>.md      # 機能設計書
  docs/screens/<slug>/list.md         # 画面設計書（一覧）
  docs/screens/<slug>/detail.md       # 画面設計書（詳細）
  ...
  docs/api/<slug>/<slug>.md           # API設計書
  docs/database/<slug>/<slug>.md      # DB設計書

次のステップ:
  実装を進める場合は /implement-design を使ってください。
```

---

## 全体を通じての注意事項

- **設計書は必ずユーザーの確認を取ってからファイル保存する**。確認なしに書き込まない
- 各ステップで「これでよいですか？修正があればお知らせください」と一言添える
- 明らかに推定できる内容（論理削除・タイムスタンプカラムなど）は質問せずデフォルトとして含める
- ユーザーの回答が不明瞭な場合は推定した内容を「〇〇と解釈しました」と明示してから進める
- 複数画面の設計書は全部まとめて確認してから一括保存してよい（往復を最小化する）
